# ドキュメント管理ガイドライン

## ドキュメントの分類

### 1. リポジトリ共通ドキュメント（`docs/`）

リポジトリ全体で共有する規約やガイドラインを定義する恒久的なドキュメント。
全ライブラリで共通のルールを提供する。

| ファイル                 | 内容                                       |
| ------------------------ | ------------------------------------------ |
| `coding-standards.md`    | コーディング規約                           |
| `development-process.md` | 開発プロセス                               |
| `testing-strategy.md`    | テスト戦略、TDD手法、テスト種別ガイド      |
| `document-management.md` | ドキュメント管理ガイドライン（本ファイル） |
| `diagram-guidelines.md`  | Mermaid/ASCII図表のベストプラクティス      |

### 2. ライブラリドキュメント（`src/<library_name>/docs/`）

各ライブラリ固有の要件・設計・規約を定義するドキュメント。
`/new-project` コマンドで作成・更新される。

| ファイル                    | 内容                                                            |
| --------------------------- | --------------------------------------------------------------- |
| `library-requirements.md`   | ライブラリの目的、使用例、受け入れ条件、機能/非機能要件（LRD）  |
| `functional-design.md`      | 機能別アーキテクチャ、データモデル、ユースケース図              |
| `architecture.md`           | テクノロジースタック、技術的制約、パフォーマンス要件            |
| `repository-structure.md`   | フォルダ構成、ディレクトリの役割、ファイル配置ルール            |
| `development-guidelines.md` | ライブラリ固有のコーディング規約、命名規則、テスト規約          |
| `glossary.md`               | ドメイン用語、技術用語、英語/日本語対応表                       |

### 3. プロジェクトファイル（`src/<library_name>/docs/project.md`）

パッケージの実装計画を定義するドキュメント。
`/new-package` コマンドで自動生成され、`/new-project` コマンドで LRD 作成時に参照される。

| セクション | 内容 |
| ---------- | ---- |
| 概要 | パッケージの目的と背景 |
| 実装計画 | マイルストーンと機能一覧 |
| 技術的考慮事項 | アーキテクチャ、依存関係、制約 |

### 4. タスク管理（GitHub Issues）

特定の開発作業における「今回何をするか」を GitHub Issues で管理。
`/issue` コマンドで project.md と双方向同期。

#### Issue テンプレート

| テンプレート | 用途 |
| ------------ | ---- |
| `feature_request.yml` | 機能要望の定義 |
| `implementation_task.yml` | 実装タスクの管理（TDD チェックリスト付き） |

#### タスク管理フロー

1. `/issue @src/<library_name>/docs/project.md` でタスク分解・Issue 作成
2. `feature-implementer` で TDD ベースの実装
3. Issue のチェックボックスで進捗管理
4. 完了時に Issue をクローズ

## 管理の原則

### リポジトリ共通ドキュメント

-   全ライブラリで共通のルールを記述
-   頻繁に更新されない
-   リポジトリ全体のポリシー変更時のみ更新

### ライブラリドキュメント

-   各ライブラリの要件・設計を記述
-   ライブラリの基本設計変更時に更新
-   `/new-project` コマンドで作成・更新

### タスク管理（GitHub Issues）

-   特定の作業・変更を Issue で管理
-   `/issue` コマンドで project.md と同期
-   Issue のチェックボックスで進捗追跡
-   完了時に振り返りを記録

## 機能追加・修正時の手順

### 1. 影響分析

ライブラリドキュメント（`src/<library_name>/docs/`）への影響を確認。
変更が基本設計に影響する場合はライブラリドキュメントを更新。

### 2. Issue 作成・タスク分解

```bash
/issue @src/<library_name>/docs/project.md
```

task-decomposer がタスクを分解し、GitHub Issues を作成。

### 3. ライブラリドキュメント更新（必要な場合のみ）

変更が基本設計に影響する場合、該当する `src/<library_name>/docs/` 内のドキュメントを更新。

### 4. 実装開始

GitHub Issue のチェックリストに基づいて TDD で実装。
`feature-implementer` を使用して Red→Green→Refactor サイクルを実行。

### 5. 品質チェック

```bash
make check-all
```

## モジュールREADME.mdの管理

### 5. モジュールドキュメント（パッケージルート）

各Pythonパッケージのルートに配置するREADME.md。Claude Codeがコンテキストを自動取得するために使用。

| ファイル | 内容 |
| -------- | ---- |
| `template/src/template_package/README.md` | テンプレートパッケージの構造・設計意図 |
| `src/<library_name>/README.md` | 各ライブラリの構造・実装状況 |

### 更新トリガー

以下の変更があった場合、該当するREADME.mdを更新する:

-   新規モジュール（.pyファイル）の追加
-   既存モジュールの削除または大幅な変更
-   公開API（`__init__.py`の`__all__`）の変更
-   ディレクトリ構造の変更
-   未実装モジュールの実装完了

### 記載必須項目

-   [ ] パッケージ概要（1-2行）
-   [ ] ディレクトリ構成図
-   [ ] 各モジュールの役割（テーブル形式）
-   [ ] 公開API一覧
-   [ ] 基本的な使用例
-   [ ] 関連ドキュメントへのリンク

### コードとの同期方針

1. **コード変更時**: 対応するREADME.mdのセクションを確認・更新
2. **PR作成時**: README.mdの更新漏れがないかチェック
3. **レビュー時**: コードとREADME.mdの整合性を確認

## 注意事項

-   ドキュメントの作成・更新は段階的に行い、各段階で承認を得る
-   リポジトリ共通ドキュメント、ライブラリドキュメント、タスク管理（GitHub Issues）を混同しない
-   コード変更後は必ず `make check-all` を実施
-   モジュールREADME.mdはコード変更と同時に更新する
