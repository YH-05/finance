# VI. 残された論点（要追加議論）

## 優先度：高

### 1. 差別化の持続性（モート） ✅ 議論済み

MASのアーキテクチャは技術的にオープンソース可能。他社が追随した場合、何が残る？

→ **結論**: AIファンド単体には構造的に強いモートが存在しにくい。発想を転換し、MASを「ファンド」ではなく「**投資知識の発見・検証・蓄積を行うプラットフォーム（Knowledge Discovery Platform）**」として位置づける。A（知識モート）+ B（プラットフォーム化）+ E（データ生成ループ）の統合戦略を採用。

→ **詳細**: [`06a_moat-strategy.md`](06a_moat-strategy.md)

### 2. 投資戦略の洗練メカニズム

MASが生成する投資戦略をどのようにサイクル的に洗練・進化させるか？

→ **要議論**:

- **TUMIX型マルチエージェント討論**: Google が提案した [TUMIX](https://arxiv.org/html/2510.01279v1)（Tool-Use Mixture）のように、異なるスタイルのエージェント群（テキスト推論/コード実行/検索/ガイド付き等）が中間結果を共有し、反復的に回答を洗練するアーキテクチャ。単一モデルの巨大化ではなく「組織としての知性」で精度を高めるアプローチ
- **リーダーボード型競争的洗練**: [Numerai](https://docs.numer.ai/) のトーナメント構造（週次ラウンド→メタモデル統合→ステーキングによるインセンティブ整合）、[MixSeek Quant Insight](https://zenn.dev/gamella/articles/a624a7eb2c78f7?locale=en) の複数AIエージェントチーム間競争による信号生成、Kaggle のタイムシリーズ競技形式など
- **両アプローチの統合**: 協調的洗練（TUMIX型）と競争的洗練（リーダーボード型）をどう組み合わせるか

### 3. リスク管理フレームワーク

MAS自体のオペレーショナルリスクへの対処。

→ **要議論**: API障害時の対応、モデル更新時の品質保証、データソース障害時のフォールバック、エージェント間の障害伝播防止

### 4. ガバナンス構造

人間FM（Fund Manager）の役割の具体化。

→ **要議論**: 投資委員会の構成、AIの判断を人間がオーバーライドする基準、責任の所在、エスカレーションフロー

### 5. テクノロジーロードマップ

MASの技術的進化の方向性。

→ **要議論**: モデルのアップグレード戦略、マルチモーダル対応計画、推論コストの最適化、エージェント構成の動的調整
